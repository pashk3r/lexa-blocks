# Описание диаграмм проектирования Lexa-Blocks

**Версия:** 2.0  
**Дата:** 20.05.2026  
**Назначение:** Единый справочник для разработчиков по всем UML-диаграммам проекта.

---
## 1. [Диаграмма классов](/generated/class-diagram.png)

**Назначение:**  
Детально описывает статическую структуру системы: классы, их атрибуты, методы и связи. Является основным артефактом для разработчиков.

**Ключевые классы ядра:**
- **Game** – главный класс, управляющий циклом, состоянием, отрисовкой и событиями. Содержит композицию с `GameState`, `Renderer`, `EventHandler` и агрегацию с `BasePlugin`.
- **GameState** – хранит всё состояние игры: поле, фигуры в руке, активную фигуру, счёт, уровень, флаг `game_over`, данные вопроса. Содержит методы для размещения фигур, добавления очков, проверки тупика, работы с вопросами.
- **Board** – игровое поле (сетка 8×8). Методы: проверка размещения, размещение фигуры, поиск и удаление заполненных строк/столбцов.
- **Block** – фигура (форма, цвет, позиция). Методы отрисовки и проверки попадания курсора.
- **Renderer** – отрисовка поля, шкалы очков, оверлея (включая экраны плагина).
- **EventHandler** – обработка событий ввода (мышь) в зависимости от состояния игры.

**Классы плагина:**
- **BasePlugin** – абстрактный базовый класс для плагинов. Определяет методы `on_game_over`, `on_restart`, `draw_overlay`, `handle_click`.
- **QuizPlugin** – конкретная реализация плагина спасения. Содержит методы отрисовки различных экранов (выбор, загрузка, вопрос, результаты) и обработчики кликов.

**Модуль Api_client:**  
Представлен как класс со стереотипом `<<module>>` и статическими методами. Осуществляет вызов DeepSeek API и генерацию локального вопроса при ошибке.

**Связи:**
- Композиция между Game и GameState, Renderer, EventHandler (они не могут существовать без Game).
- Композиция между GameState и Board.
- Агрегация между GameState и Block (фигуры в руке могут быть заменены).
- Зависимость QuizPlugin от модуля api_client (через вызовы).
- QuizPlugin использует перечисление `GameOverlayState` для констант состояний.

**Примечание:**  
Классы содержат как публичные (`+`), так и приватные (`-`) методы. Приватные методы (с подчёркиванием) предназначены для внутреннего использования.

## 2. [Компонентная диаграмма](/generated/component-diagram.png)

**Назначение:**  
Показывает высокоуровневую архитектуру системы: основные компоненты и их зависимости.

**Компоненты:**
- **GameCore** – ядро игры (классы: `Game`, `GameState`, `Board`, `Block`, `Renderer`, `EventHandler`). Отвечает за основную механику: поле, фигуры, очки, уровни, отрисовку и ввод.
- **QuizPlugin** – плагин спасения, наследуется от `BasePlugin`. Реализует логику вопросов, взаимодействие с API и выдачу спасательных кубов.
- **External API Client** – модуль `api_client`, предоставляющий функции `fetch_question_async` и `make_error_question`.
- **DeepSeek API** – внешний сервис, вызываемый по HTTP.

**Зависимости:**
- `GameComp` использует `QuizPluginComp`.
- `QuizPluginComp` наследует `BasePluginComp` (расширение функциональности).
- `QuizPluginComp` вызывает модуль `api_client` для загрузки вопроса.
- `api_client` отправляет HTTP-запросы к DeepSeek API.

## 3. [Диаграмма последовательности: Спасение через математический вопрос (UC-4)](/generated/sequence_loss_and_math.png)

**Назначение:**  
Описывает сценарий, когда игрок после окончания игры выбирает «Ответить на вопрос». Показывает взаимодействие с внешним API (DeepSeek), обработку ответа и выдачу спасательных кубов.

**Участники:**
- **Игрок**
- **Game** – главный игровой цикл.
- **GameState** – состояние игры (флаги, вопрос, варианты).
- **QuizPlugin** – плагин, реализующий логику спасения.
- **Api_client** – модуль для асинхронного вызова API.
- **DeepSeek API** – внешний сервис генерации вопросов.

**Сценарий:**
1. Игра переходит в состояние `game_over = True`, вызывается метод плагина `on_game_over`.
2. Плагин устанавливает состояние `STATE_CHOICE`, игрок видит кнопку «Ответить на вопрос».
3. При клике на кнопку плагин переводит состояние в `STATE_LOADING` и вызывает `fetch_question_async`.
4. Api_client отправляет HTTP-запрос к DeepSeek API и через колбэк возвращает вопрос, варианты и индекс правильного ответа.
5. Плагин сохраняет данные вопроса в GameState и переходит в `STATE_QUESTION`.
6. Игрок видит вопрос и 4 варианта, выбирает один.
7. **Если ответ правильный:** плагин показывает экран «Правильно!», затем сбрасывает флаг `game_over`, вызывает `spawn_rescue_blocks()` (выдаёт три одиночных куба), очищает данные вопроса и возвращается в обычное состояние.
8. **Если ответ неправильный:** плагин показывает «Неверно!», затем полностью пересоздаёт GameState (`__init__`), что приводит к полному сбросу игры (счёт=0, поле очищено).

**Примечание:**  
При ошибке API вызывается колбэк `on_error`, который генерирует локальный вопрос с вариантами «Повторить/Сдаться», позволяя игроку повторить попытку или сдаться.

## 4. [Диаграмма последовательности: Размещение фигуры (UC-1)](/generated/sequence_place_figure.png)

**Назначение:**  
Описывает основной геймплейный сценарий – игрок перетаскивает фигуру из слота на поле, система проверяет возможность размещения, очищает заполненные линии, начисляет очки и при необходимости генерирует новый набор фигур.

**Участники:**
- **Игрок** – пользователь.
- **GameState** – состояние игры (логика размещения).
- **Board** – игровое поле.

**Сценарий:**
1. Игрок перетаскивает фигуру и отпускает над клеткой поля.
2. GameState вызывает `try_place_active_block()`, который:
   - Вычисляет координаты (gx, gy) по позиции курсора.
   - Запрашивает у Board проверку `can_place`.
3. **Если размещение возможно:**
   - Board выполняет `place` (фигура записывается в сетку, затем вызывается `_clear_lines` для удаления заполненных строк/столбцов).
   - Board возвращает количество начисленных очков (1 очко за каждый куб фигуры + 10 очков за каждую очищенную линию).
   - GameState добавляет очки к счёту (`_add_score`).
   - GameState удаляет использованную фигуру из списка `blocks_in_hand`.
   - Если в руке не осталось фигур, GameState генерирует новый набор (`spawn_blocks`).
   - GameState проверяет, есть ли возможные ходы с новым набором (`_is_game_over`). Если нет – устанавливает `game_over = True`.
4. **Если размещение невозможно:**
   - GameState возвращает фигуру на исходную позицию в слоте.

**Ключевые моменты:**
- Очки начисляются в два этапа: сначала за кубы, потом за линии. Это важно для правильного подсчёта (в коде они суммируются в `place`).
- Генерация нового набора происходит только после того, как игрок использовал все три фигуры, а не после каждого хода (что типично для игр типа Block Blast).

## 5. [Диаграмма последовательности: Использование спасательного куба (UC-5)](/generated/sequence_use_save_cube.png)

**Назначение:**  
Показывает процесс использования игроком спасательного куба (одиночная клетка 1×1) после того, как он был получен за правильный ответ на вопрос. Демонстрирует размещение куба, начисление очков и возможную последующую генерацию новых фигур.

**Участники:**
- **Игрок** – пользователь, перетаскивающий куб.
- **GameState** – состояние игры (содержит логику размещения, фигуры в руке, счёт).
- **Board** – игровое поле, отвечающее за проверку и размещение.

**Сценарий:**
1. Игрок перетаскивает одиночный куб на клетку поля.
2. GameState запрашивает у Board проверку возможности размещения.
3. **Если ячейка свободна:**
   - Board размещает куб, удаляет заполненные линии и возвращает начисленные очки (1 очко за куб + очки за линии).
   - GameState добавляет очки к общему счёту.
   - GameState удаляет использованный куб из списка фигур в руке.
   - Если фигуры в руке закончились, GameState генерирует новый набор обычных фигур.
   - GameState проверяет наличие возможных ходов (если нет – игра заканчивается).
4. **Если ячейка занята:** куб возвращается на исходную позицию в слоте.

**Ключевые решения:**  
Спасательный куб ничем не отличается от обычной фигуры по механике размещения, но он всегда имеет форму 1×1 и даёт 1 очко за размещение.

## 6. [Диаграмма состояний](/generated/state-machine-diagram.png)

**Назначение:**  
Описывает конечный автомат игры, все возможные состояния оверлея (экрана после окончания игры) и переходы между ними.

**Состояния и переходы:**
- **`STATE_GAME_OVER`** – начальное состояние при запуске игры или после рестарта. При вызове `on_game_over()` плагина переход в `STATE_CHOICE`.
- **`STATE_CHOICE`** – экран выбора: кнопки «Начать заново» и «Ответить на вопрос».  
  - При выборе «Начать заново» → возврат в `STATE_GAME_OVER` с полным сбросом.  
  - При выборе «Ответить на вопрос» → переход в `STATE_LOADING`.
- **`STATE_LOADING`** – отображает загрузку, выполняется асинхронный запрос к API.  
  - При успешном ответе → `STATE_QUESTION`.  
  - При ошибке API → также `STATE_QUESTION`, но с локальным вопросом-заглушкой.
- **`STATE_QUESTION`** – показ вопроса и четырёх вариантов ответа. Игрок выбирает вариант.  
  - Правильный вариант → `STATE_RESULT_OK`.  
  - Неправильный вариант → `STATE_RESULT_FAIL`.
- **`STATE_RESULT_OK`** – сообщение «Правильно! Нажми, чтобы продолжить». По клику → `STATE_GAME_OVER` (с выдачей спасательных кубов и снятием флага game_over).
- **`STATE_RESULT_FAIL`** – сообщение «Неверно! Нажми, чтобы начать заново». По клику → полный сброс состояния (вызов `__init__`).
