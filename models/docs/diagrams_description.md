# Описание диаграмм проектирования Lexa-Blocks

**Версия:** 3.0  
**Дата:** 23.05.2026  
**Назначение:** Единый справочник для разработчиков по всем UML-диаграммам проекта.

---

## 1. [Диаграмма классов](/generated/class-diagram.png)

**Назначение:**  
Детально описывает статическую структуру системы: классы, их атрибуты, методы и связи. Является основным артефактом для разработчиков.

**Ключевые классы ядра:**
- **Game** – главный класс, управляющий циклом, состоянием, отрисовкой и событиями. Содержит композицию с `GameState`, `Renderer`, `EventHandler` и агрегацию с `BasePlugin`.
- **GameState** – хранит всё состояние игры: поле, фигуры в руке, активную фигуру, счёт, уровень, флаг `game_over`, а также данные для трёх типов вопросов:
  - `question_text`, `options`, `correct_index`, `selected_index` – для типа "выбор варианта".
  - `input_text`, `correct_answer` – для типа "ввод ответа".
  - `error_steps`, `error_step_index` – для типа "найди ошибку".
  Содержит методы для размещения фигур, добавления очков, проверки тупика, а также специализированные методы `apply_choice_question`, `apply_input_question`, `apply_find_error_question` и `clear_question`.
- **Board** – игровое поле (сетка 8×8). Методы: проверка размещения, размещение фигуры, поиск и удаление заполненных строк/столбцов.
- **Block** – фигура (форма, цвет, позиция). Методы отрисовки и проверки попадания курсора.
- **Renderer** – отрисовка поля, шкалы очков, оверлея (включая экраны плагина).
- **EventHandler** – обработка событий ввода (мышь) в зависимости от состояния игры. Также поддерживает клавиатурный ввод для состояния `STATE_INPUT`.

**Классы плагина:**
- **BasePlugin** – абстрактный базовый класс для плагинов. Определяет методы `on_game_over`, `on_restart`, `draw_overlay`, `handle_click`. Опционально может определять `handle_keydown`.
- **QuizPlugin** – конкретная реализация плагина спасения. Поддерживает три типа викторины:
  - `choice` – вопрос с 4 вариантами ответа.
  - `input` – вопрос с полем ввода текста/числа.
  - `find_error` – фрагмент кода, в котором нужно найти ошибочный шаг.
  Содержит методы отрисовки всех экранов (`_draw_choice`, `_draw_loading`, `_draw_question`, `_draw_input`, `_draw_find_error`, `_draw_result_ok`, `_draw_result_fail`) и обработчики кликов/клавиатуры.

**Модуль api_client:**  
Представлен как класс со стереотипом `<<module>>` и статическими методами. Осуществляет асинхронный вызов DeepSeek API с параметром `quiz_type` (choice/input/find_error) и генерацию локального вопроса при ошибке через `make_error_question`.

**Связи:**
- Композиция между Game и GameState, Renderer, EventHandler.
- Композиция между GameState и Board.
- Агрегация между GameState и Block.
- Зависимость QuizPlugin от api_client.
- QuizPlugin использует перечисления `GameOverlayState` и `QuizType` для констант.

**Примечание:**  
Классы содержат как публичные (`+`), так и приватные (`-`) методы. Приватные методы (с подчёркиванием) предназначены для внутреннего использования.

## 2. [Компонентная диаграмма](/generated/component-diagram.png)

**Назначение:**  
Показывает высокоуровневую архитектуру системы: основные компоненты и их зависимости.

**Компоненты:**
- **GameCore** – ядро игры (классы: `Game`, `GameState`, `Board`, `Block`, `Renderer`, `EventHandler`). Отвечает за основную механику: поле, фигуры, очки, уровни, отрисовку и ввод.
- **QuizPlugin** – плагин спасения, наследуется от `BasePlugin`. Реализует три типа викторины, взаимодействие с API и выдачу спасательных кубов.
- **External API Client** – модуль `api_client`, предоставляющий функции `fetch_question_async(quiz_type, ...)` и `make_error_question(error_msg, quiz_type)`.
- **DeepSeek API** – внешний сервис, вызываемый по HTTP с различными промптами в зависимости от типа вопроса.

**Зависимости:**
- `GameCore` использует `QuizPlugin` (через механизм плагинов).
- `QuizPlugin` наследует `BasePlugin`.
- `QuizPlugin` вызывает модуль `api_client` для загрузки вопроса.
- `api_client` отправляет HTTP-запросы к DeepSeek API, передавая соответствующий промпт.

## 3. [Диаграмма последовательности: Спасение через викторину (UC-4)](/generated/sequence_loss_and_math.png)

**Назначение:**  
Описывает сценарий, когда игрок после окончания игры выбирает один из трёх типов викторины (choice, input, find_error) или случайный тип. Показывает асинхронное взаимодействие с DeepSeek API, обработку ответа и выдачу спасательных кубов.

**Участники:**
- **Игрок**
- **Game** – главный игровой цикл.
- **GameState** – состояние игры (флаги, вопрос, варианты, введённый текст, шаги кода).
- **QuizPlugin** – плагин, реализующий логику всех трёх типов.
- **api_client** – модуль для асинхронного вызова API.
- **DeepSeek API** – внешний сервис генерации вопросов.

**Сценарий (обобщённый):**
1. При `game_over = True` вызывается `on_game_over()` плагина, который переводит состояние в `STATE_CHOICE`. Игрок видит кнопки: «Начать заново», «Варианты ответа», «Введи ответ», «Найди ошибку», «Случайный вопрос».
2. При выборе типа викторины плагин переходит в `STATE_LOADING` и вызывает `fetch_question_async(quiz_type, ...)`.
3. api_client отправляет HTTP-запрос к DeepSeek API с соответствующим промптом.
4. API возвращает JSON, который парсится в зависимости от типа.
5. Плагин сохраняет данные в GameState через один из методов:
   - `apply_choice_question` – для choice (сохраняет вопрос, варианты, правильный индекс).
   - `apply_input_question` – для input (сохраняет вопрос и правильный ответ).
   - `apply_find_error_question` – для find_error (сохраняет список шагов и индекс ошибки).
6. Плагин переводит состояние в `STATE_QUESTION`, `STATE_INPUT` или `STATE_FIND_ERROR`.
7. Игрок взаимодействует:
   - Для choice – выбирает один из 4 вариантов.
   - Для input – вводит текст (поддерживается клавиатура) и нажимает «Подтвердить».
   - Для find_error – кликает по одному из шагов кода.
8. Плагин проверяет ответ:
   - Если ответ правильный → переход в `STATE_RESULT_OK`.
   - Если неправильный → переход в `STATE_RESULT_FAIL`.
9. В состоянии `STATE_RESULT_OK` игрок кликает для продолжения; плагин снимает флаг `game_over`, вызывает `spawn_rescue_blocks()` (выдаёт три одиночных куба), очищает данные вопроса и возвращается в `STATE_GAME_OVER`.
10. В состоянии `STATE_RESULT_FAIL` игрок кликает; плагин полностью пересоздаёт GameState (`__init__`), сбрасывая счёт, поле и кубы.

**Примечание:**  
При ошибке API вызывается `on_error`, который через `make_error_question` генерирует локальный вопрос соответствующего типа, позволяя игроку повторить или сдаться.

## 4. [Диаграмма последовательности: Размещение фигуры (UC-1)](/generated/sequence_place_figure.png)

**Назначение:**  
Описывает основной геймплейный сценарий – игрок перетаскивает фигуру из слота на поле, система проверяет возможность размещения, очищает заполненные линии, начисляет очки и при необходимости генерирует новый набор фигур.

**Участники:**
- **Игрок** – пользователь.
- **GameState** – состояние игры (логика размещения).
- **Board** – игровое поле.

**Сценарий:**
1. Игрок перетаскивает фигуру и отпускает над клеткой поля.
2. GameState вызывает `try_place_active_block()`, который:
   - Вычисляет координаты (gx, gy) по позиции курсора.
   - Запрашивает у Board проверку `can_place`.
3. **Если размещение возможно:**
   - Board выполняет `place` (фигура записывается в сетку, затем вызывается `_clear_lines` для удаления заполненных строк/столбцов).
   - Board возвращает количество начисленных очков (1 очко за каждый куб фигуры + 10 очков за каждую очищенную линию).
   - GameState добавляет очки к счёту (`_add_score`).
   - GameState удаляет использованную фигуру из списка `blocks_in_hand`.
   - Если в руке не осталось фигур, GameState генерирует новый набор (`spawn_blocks`).
   - GameState проверяет, есть ли возможные ходы с новым набором (`_is_game_over`). Если нет – устанавливает `game_over = True`.
4. **Если размещение невозможно:**
   - GameState возвращает фигуру на исходную позицию в слоте.

**Ключевые моменты:**
- Очки начисляются в два этапа: сначала за кубы, потом за линии.
- Генерация нового набора происходит только после того, как игрок использовал все три фигуры.

## 5. [Диаграмма последовательности: Использование спасательного куба (UC-5)](/generated/sequence_use_save_cube.png)

**Назначение:**  
Показывает процесс использования игроком спасательного куба (одиночная клетка 1×1) после того, как он был получен за правильный ответ на вопрос.

**Участники:**
- **Игрок**
- **GameState**
- **Board**

**Сценарий:**
1. Игрок перетаскивает одиночный куб на клетку поля.
2. GameState запрашивает у Board проверку возможности размещения.
3. **Если ячейка свободна:**
   - Board размещает куб, удаляет заполненные линии и возвращает начисленные очки (1 очко за куб + очки за линии).
   - GameState добавляет очки к общему счёту.
   - GameState удаляет использованный куб из списка фигур в руке.
   - Если фигуры в руке закончились, GameState генерирует новый набор обычных фигур.
   - GameState проверяет наличие возможных ходов (если нет – игра заканчивается).
4. **Если ячейка занята:** куб возвращается на исходную позицию в слоте.

**Ключевые решения:**  
Спасательный куб – это обычная фигура формы 1×1. После правильного ответа игрок получает три таких куба, которые добавляются в руку.

## 6. [Диаграмма состояний](/generated/state-machine-diagram.png)

**Назначение:**  
Описывает конечный автомат игры, все возможные состояния оверлея и переходы между ними, включая три типа викторины.

**Состояния и переходы:**
- **`STATE_GAME_OVER`** – начальное состояние при запуске игры или после рестарта. При вызове `on_game_over()` плагина переход в `STATE_CHOICE`.
- **`STATE_CHOICE`** – экран выбора: кнопки «Начать заново», «Варианты ответа» (choice), «Введи ответ» (input), «Найди ошибку» (find_error), «Случайный вопрос».  
  - При выборе «Начать заново» → возврат в `STATE_GAME_OVER` с полным сбросом.  
  - При выборе любого другого типа → переход в `STATE_LOADING`.
- **`STATE_LOADING`** – отображает загрузку, выполняется асинхронный запрос к API с соответствующим `quiz_type`. После получения ответа:
  - Если тип choice → переход в `STATE_QUESTION`.
  - Если тип input → переход в `STATE_INPUT`.
  - Если тип find_error → переход в `STATE_FIND_ERROR`.
- **`STATE_QUESTION`** – показ вопроса с 4 вариантами. Игрок выбирает вариант.  
  - Правильный → `STATE_RESULT_OK`.  
  - Неправильный → `STATE_RESULT_FAIL`.
- **`STATE_INPUT`** – показ вопроса и поля ввода. Игрок вводит текст (клавиатура) и подтверждает.  
  - Правильный ответ → `STATE_RESULT_OK`.  
  - Неправильный → `STATE_RESULT_FAIL`.
- **`STATE_FIND_ERROR`** – показ списка шагов кода. Игрок кликает по одному из шагов.  
  - Выбран ошибочный шаг → `STATE_RESULT_OK`.  
  - Выбран не ошибочный → `STATE_RESULT_FAIL`.
- **`STATE_RESULT_OK`** – сообщение «Правильно! Нажми, чтобы продолжить». По клику → `STATE_GAME_OVER` (с выдачей спасательных кубов и снятием флага game_over).
- **`STATE_RESULT_FAIL`** – сообщение «Неверно! Нажми, чтобы начать заново». По клику → полный сброс состояния (вызов `__init__`).

**Особенности:**
- Во время `STATE_INPUT` обрабатываются клавиши Backspace, Enter, а также печатаемые символы.
- При ошибке API в любом из типов автоматически генерируется локальный вопрос соответствующего типа, чтобы игра не зависала.
