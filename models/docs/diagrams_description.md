# Описание диаграмм проектирования Lexa-Blocks

**Версия:** 1.0  
**Дата:** 14.05.2026  
**Назначение:** Единый справочник для разработчиков по всем UML-диаграммам проекта.

---

## 1. Компонентная диаграмма (`models/component-diagram.puml`)

### Что показывает
- Крупные модули (компоненты) системы и интерфейсы между ними.
- Основные модули: `GameCore` (игровая логика), `EduModule` (образовательная часть – генерация вопросов).
- Интерфейс `IEduService` – контракт между игровым ядром и модулем вопросов.
- Зависимость `EduModule` от внешнего API (облачный сервис).

### Ключевые элементы
- **GameCore** – отвечает за поле 8×8, фигуры, очистку линий, очки, уровни, спасательные кубы, конечный автомат игры.
- **EduModule** – реализует `IEduService`, скрывает детали генерации вопросов (локально или через API).
- **IEduService** – методы `generateQuestion(): Question` и `verifyAnswer(id, option): bool`.
- **Remote AI API** – внешний сервис, вызываемый `EduModule` по HTTP (JSON).

### Для разработчиков
- При написании кода **GameCore не должен напрямую обращаться к API вопросов** – только через `IEduService`.
- Это позволяет легко заменять реализацию `EduModule` (мок для тестов, локальный генератор, другой провайдер).
- Связь `GameCore ..> IEduService` – это dependency injection (интерфейс внедряется в GameCore).

## 2. Диаграмма классов (`models/class-diagram.puml`)

### Что показывает
- Статическую структуру: классы, их поля, методы, видимость, связи (наследование, композиция, реализация интерфейсов).
- Все сущности предметной области: `Field`, `Cell`, `Figure`, `ScoreSystem`, `SaveCubesManager`, `GameManager`, `IEduService`, `RemoteMathAPI`, `MathQuestion`.

### Ключевые элементы
| Класс | Ответственность | Примечание |
|-------|----------------|------------|
| `GameManager` | Координация игры, состояния, обработка ходов, тупика, спасения | Содержит ссылки на `Field`, `ScoreSystem`, `SaveCubesManager`, `IEduService` |
| `Field` | Сетка 8×8, проверка и размещение фигур, очистка строк/столбцов | Метод `clearFullRowsAndColumns()` возвращает количество очищенных линий |
| `ScoreSystem` | Подсчёт очков, уровня, прогресса | `addPointsForCubes(cubes)` – 1 очко за куб; `addPointsForLines(lines)` – 10 очков за линию; `getLevel()` = score//100 + 1; `getProgress()` = score % 100 |
| `SaveCubesManager` | Управление спасательными кубами (без лимита) | `add(3)`, `use(): bool` |
| `Figure` | Форма фигуры (список относительных координат) | Предопределённые формы: 2×2, линии, L, одиночный куб |
| `IEduService` | Интерфейс образовательного модуля | Две операции: генерация вопроса, проверка ответа |
| `RemoteMathAPI` | Реализация `IEduService` через внешний API | Содержит `apiBaseUrl`, `timeout` |
| `MathQuestion` | DTO для вопроса | id, текст, список вариантов, правильный вариант (A/B/C/D) |
| `GameState` | Перечисление состояний | `PLAYING`, `LOSS_MENU`, `WAITING_MATH_ANSWER` |

### Связи
- `GameManager *-- Field` – композиция (поле не живёт без менеджера).
- `GameManager *-- ScoreSystem` – композиция.
- `GameManager *-- SaveCubesManager` – композиция.
- `GameManager --> IEduService` – зависимость (интерфейс).
- `RemoteMathAPI ..|> IEduService` – реализация интерфейса.
- `RemoteMathAPI --> MathQuestion` – генерирует объект вопроса.

### Для разработчиков
- Следуйте именам методов и типам, указанным на диаграмме.
- Композиция (`*--`) означает, что время жизни дочернего объекта совпадает с временем жизни родителя (при удалении `GameManager` всё внутреннее тоже удаляется).
- Интерфейс `IEduService` должен быть реализован в отдельном классе (или модуле), который можно внедрять в `GameManager` через конструктор или сеттер.

## 3. Диаграмма состояний (`models/state-machine.puml`)

### Что показывает
- Конечный автомат игры: все возможные состояния и переходы между ними.
- Состояния: `PLAYING`, `LOSS_MENU`, `WAITING_MATH_ANSWER` (с подсостояниями).
- Действия, вызывающие переходы (события, условия).

### Ключевые состояния и переходы

| Текущее состояние | Событие | Следующее состояние | Действие |
|-------------------|---------|---------------------|----------|
| `[*]` | start game | `PLAYING` | – |
| `PLAYING` | нет возможных ходов (тупик) | `LOSS_MENU` | показать меню с кнопками Restart / Math Question |
| `LOSS_MENU` | Restart | `PLAYING` | полный сброс игры |
| `LOSS_MENU` | Math Question | `WAITING_MATH_ANSWER` | начать генерацию вопроса |
| `WAITING_MATH_ANSWER` | вопрос получен | `DisplayQuestion` | отобразить вопрос и варианты |
| `WAITING_MATH_ANSWER` | ошибка API | `RetryOrGiveUp` | показать кнопки Повторить / Сдаться |
| `DisplayQuestion` | правильный ответ | `PLAYING` | добавить 3 спасательных куба, продолжить игру |
| `DisplayQuestion` | неправильный ответ или Сдаться | `PLAYING` (сброс) | полный сброс (счёт=0, поле пусто, кубы=0) |
| `RetryOrGiveUp` | Повторить | `GeneratingQuestion` | повторная попытка запроса |
| `RetryOrGiveUp` | Сдаться | `PLAYING` (сброс) | полный сброс |

### Для разработчиков
- В коде должно быть перечисление `GameState` (или отдельный класс-автомат).
- Действия при переходах (например, добавление кубов, сброс) должны выполняться строго в момент перехода.

## 4. Диаграммы последовательности

### 4.1. Размещение фигуры и очистка линий (`sequence_place_figure.puml`)

**Сценарий:** UC-1 + UC-2 (успешное размещение фигуры, очистка строк/столбцов, начисление очков, проверка тупика).

**Поток:**
1. Игрок перетаскивает фигуру и отпускает над полем.
2. `GameManager` проверяет `Field.canPlace()`.
3. Если можно – `Field.placeFigure()`.
4. `ScoreSystem.addPointsForCubes()` (1 очко за куб).
5. `Field.clearFullRowsAndColumns()` → возвращает количество очищенных линий.
6. `ScoreSystem.addPointsForLines(lines)` (10 очков за линию).
7. Удалить размещённую фигуру из панели (`currentFigures`).
8. Если панель пуста – сгенерировать новый набор из 3 фигур.
9. `GameManager.checkGameOver()` – если нет возможных ходов, перейти в `LOSS_MENU`.

**Для разработчиков:**  
Реализуйте методы как в диаграмме. Обратите внимание, что очки начисляются **сначала за кубы**, затем **за линии**.

### 4.2. Тупик и спасение через вопрос (`sequence_loss_and_math.puml`)

**Сценарий:** UC-4 (игрок в тупике, выбирает «Ответить на вопрос», взаимодействие с API).

**Поток:**
1. `GameManager` переходит в `LOSS_MENU`.
2. Игрок нажимает «Ответить на вопрос».
3. `GameManager` переходит в `WAITING_MATH_ANSWER`, вызывает `IEduService.generateQuestion()`.
4. **При успешном ответе API** → отображается вопрос с 4 вариантами.
5. Игрок выбирает вариант (или нажимает «Сдаться»).
   - «Сдаться» → `fullReset()`.
   - Выбран вариант → `IEduService.verifyAnswer()`.
6. **Правильный ответ** → `saveCubesManager.add(3)`, переход в `PLAYING`.
7. **Неправильный ответ** → `fullReset()`.
8. **При ошибке API** → диалог «Повторить / Сдаться»; при повторе цикл повторяется, при сдаче – `fullReset()`.

**Для разработчиков:**  
- `IEduService` должен быть асинхронным (возвращать Promise/Future) из-за сетевых вызовов.
- Не блокируйте UI во время ожидания ответа API.
- Обрабатывайте все исключения: таймаут, 5xx, некорректный JSON.

### 4.3. Использование спасательного куба (`sequence_use_save_cube.puml`)

**Сценарий:** UC-5 (игрок имеет кубы, размещает одиночный куб на поле).

**Поток:**
1. Игрок выбирает спасательный куб (отдельный режим или специальная кнопка).
2. Кликает на свободную ячейку.
3. `GameManager` вызывает `SaveCubesManager.use()` (уменьшает счётчик).
4. Создаётся временная фигура 1×1 и размещается через `Field.placeFigure()`.
5. Начисляются очки за куб (1 очко).
6. `Field.clearFullRowsAndColumns()` – возможно, очистка строк/столбцов после размещения куба.
7. Начисляются очки за линии (если есть).
8. Обновляется UI.

**Для разработчиков:**  
- Счётчик кубов хранится в `SaveCubesManager` (начальное значение 0, увеличивается только через `add()`).
- Размещение куба ничем не отличается от размещения обычной фигуры, кроме того, что форма = одиночная клетка.
- Кнопка использования куба должна быть активна только при `saveCubes > 0` и в состоянии `PLAYING`.

- Разработчики читают эти описания вместе с диаграммами (PNG или PlantUML).
