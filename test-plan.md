**Версия:** 1.0  
**Дата:** 14-05-2026  
**Тестировщик:** [egorka000]  
**Ветка:** `testing`

---

## 1. Введение

Данный документ определяет стратегию, цели, объём и подходы к тестированию игры **Lexa-Blocks** — блок-головоломки на Python/pygame с образовательным элементом (математические вопросы при проигрыше).

Тестирование проводится на основе:
- SRS v1.0 (требования G1–G9)
- Use Cases UC-1 — UC-5
- Диаграммы состояний (state-machine)
- Кода из ветки `development` (коммит `7298517`)

---

## 2. Цели тестирования

| № | Цель |
|---|------|
| 1 | Проверить игровую механику: размещение фигур, очистка линий, начисление очков |
| 2 | Проверить переходы между состояниями игры |
| 3 | Проверить работу плагина викторины (запрос, ответ, сброс/продолжение) |
| 4 | Проверить систему спасательных кубов |
| 5 | Проверить граничные и негативные сценарии |
| 6 | Зафиксировать расхождения реализации с требованиями SRS |

---

## 3. Объём тестирования (Scope)

### В объёме (In Scope)

- Игровое поле (размер, начальное состояние)
- Размещение 5 типов фигур из SRS G2
- Панель из 3 фигур и генерация нового набора
- Очистка строк и столбцов
- Система очков: за линию + за кубы фигуры
- Уровни и прогресс (1 очко = 1%, каждые 100 = новый уровень)
- Состояния: `PLAYING`, `LOSS_MENU` (`STATE_CHOICE`), `WAITING_MATH_ANSWER` (`STATE_QUESTION`)
- Условие тупика
- Математический вопрос через DeepSeek API (мокируется в тестах)
- Правильный ответ: +3 спасательных куба, игра продолжается
- Неправильный ответ / «Начать заново»: полный сброс (счёт=0, уровень=1, поле пустое, кубы=0)
- Использование спасательного куба
- Обработка ошибки соединения с API

### Вне объёма (Out of Scope)

- Тестирование самого DeepSeek API (внешняя зависимость, мокируется)
- Визуальное/UI тестирование (pygame-рендеринг)
- Нагрузочное и производительное тестирование

---

## 4. Виды тестирования

| Вид | Что проверяем | Инструмент |
|-----|---------------|------------|
| **Модульное (Unit)** | `board.py`, `game_state.py`, `quiz_plugin.py` по отдельности | pytest |
| **Интеграционное** | Взаимодействие `GameState` + `QuizPlugin` + `api_client` | pytest + unittest.mock |
| **Негативное** | Граничные случаи, недоступный API, неверные позиции | pytest |

## 7. Приоритеты дефектов

| Приоритет | Описание | Примеры |
|-----------|----------|---------|
| **P1 — Критический** | Игра падает или ключевая функция полностью сломана | Краш при размещении, game over не срабатывает |
| **P2 — Высокий** | Функция работает неверно | Неверное начисление очков, неправильный переход состояний |
| **P3 — Средний** | Отклонение от требований, но некритично | Неверный прогресс-бар |
| **P4 — Низкий** | Косметика | Опечатки в тексте |

---

## 8. Риски

| Риск | Митигация |
|------|-----------|
| DeepSeek API недоступен во время тестов | Мокировать через `unittest.mock.patch` |
| Реализация расходится с SRS (найдено: формула очков) | Зафиксировать как дефект в `test-report.md` |
| Структура классов отличается от диаграммы | Тесты пишутся под реальный код, а не под диаграмму |
